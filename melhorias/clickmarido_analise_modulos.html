⚠️ Gaps Críticos do Sistema Atual

Funcionalidades ausentes que impactam operações do dia a dia.

1. Integração Backend Incompleta
Os módulos comentados no app.module.ts não estão implementados. O frontend chama rotas que não existem no NestJS. Ação: Uncomment e criar os serviços.
2. Sem Agendamento de Visitas
Não há modelo Appointment. Impossível planejar cronograma de técnicos. Crítico para operação de reparos residenciais.
3. Sem Controle de Inventário
Módulo de materiais vazio. Impossível rastrear consumo de insumos, custos reais, reposição automática.
4. Sem Integração WhatsApp/SMS
Conversas devem ser internas. Sem API integrada para comunicação com clientes via WhatsApp/SMS.
5. Dashboard Vazio
Rota /dashboard existe mas sem KPIs visuais. Sem gráficos de receita, ordens abertas, performance de técnicos.
6. Sem Fotos/Evidências
Nenhum modelo para storage de imagens antes/depois. Crítico para garantias e justificativa de trabalho.
7. Sem Integração Bancária/PIX
Pagamentos são apenas registros. Sem confirmação automática via PIX, extratos bancários, conciliação.
8. Sem Alertas/Notificações
Sem sistema de notificações: técnico não recebe aviso de nova OS, cliente não sabe quando técnico chega.
9. Sem Assinatura Digital/Aceite
Cliente não assina orçamento. Sem prova de entrega do trabalho no local.
10. Auditoria Desativada
Interceptors de logging/auditoria comentados. Impossível rastrear alterações críticas.

🚀 Sugestões de Novos Módulos e Funcionalidades

Priorizado por impacto operacional e viabilidade. Baseado em padrões de negócio de serviços residenciais.

CRÍTICO Semana 1-2

1. Agendamento de Visitas (Appointments)

Por quê: Sem agenda, impossível organizar rota de técnicos. Operação fica manual e ineficiente.

O que implementar:

  • Modelo Appointment: date, time, duration, status (SCHEDULED, CONFIRMED, IN_PROGRESS, COMPLETED, CANCELLED), relation a ServiceOrder + Technician
  • View Calendário: Visualização semanal/mensal com drag-drop de agendamentos
  • Notificações: Email/SMS para técnico 1h antes de começar
  • Geolocalização: Rastrear técnico durante visita (já tem GeolocationModule no backend)
  • Conflito Detection: Aviso se técnico tem duas visitas sobrepostas

Esforço: 20-30 horas | Banco: 1 novo modelo + 3 migrations

CRÍTICO Semana 1-2

2. Gerenciamento de Fotos/Evidências

Por quê: Documentação visual é prova de trabalho executado. Essencial para garantias e disputas.

O que implementar:

  • Modelo Media: id, serviceOrderId, fileUrl, type (BEFORE, AFTER, PROGRESS), uploadedAt, uploadedByUserId
  • Upload com UI: Drag-drop na página de OS, pré-visualização, compressão cliente-side
  • Storage: Usar AWS S3 (Vercel + S3) ou GCP Cloud Storage (zero-budget: Google Drive API)
  • Galeria: Timeline de fotos antes/depois com timestamps
  • Mobile: Câmera nativa via PWA para app mobile

Esforço: 18-25 horas | Banco: 1 novo modelo

CRÍTICO Semana 2-3

3. Dashboard com KPIs Visuais

Por quê: Visão consolidada de métricas permite tomada de decisão rápida. Sem dados visuais, você opera cego.

O que implementar:

  • Cards: Receita mês, ordens abertas, técnicos disponíveis, satisfação média
  • Gráficos: Receita por semana (linha), tipos de serviço (pizza), performance técnico (barras)
  • Filtros: Por período, técnico, tipo de serviço
  • Biblioteca: Usar Recharts (já no package.json) ou Chart.js
  • Dados em tempo real: Usar SWR/TanStack Query para refresh periódico

Esforço: 15-20 horas | Banco: Apenas queries otimizadas no backend

ALTO Semana 3-4

4. Integração WhatsApp / SMS para Notificações

Por quê: Comunicação automática reduz no-show de clientes. WhatsApp é ubíquo no Brasil.

O que implementar:

  • Provider abstrato: Interface que suporta múltiplos provedores (Twilio, AWS SNS, Vonage)
  • Triggers automáticos: Enviar SMS quando OS é criada, 1h antes de appointment, quando pagamento é recebido
  • Templates: Mensagens customizáveis por empresa
  • Log de mensagens: Modelo para rastrear quem recebeu o quê
  • Sugestão zero-cost: Usar Google Cloud Text-to-Speech + Twilio free trial (100 SMS grátis/mês)

Esforço: 25-35 horas | Banco: 1-2 novos modelos (Message, MessageLog)

ALTO Semana 4-5

5. Controle de Inventário (Materiais)

Por quço: Rastreamento de gastos de materiais permite precificação correta e margem real.

O que implementar:

  • Modelos: Material (name, category, costPrice, quantity, minStock, supplier), MaterialUsage (serviceOrderId, materialId, quantityUsed)
  • Alertas: Notificar quando estoque < minStock
  • Integração com OS: Ao completar OS, registrar materiais consumidos
  • Relatório: Custo de material por OS, ticket médio incluindo material
  • Importação: CSV para carga inicial de inventário

Esforço: 20-25 horas | Banco: 2 novos modelos + constraints

ALTO Semana 2-3

6. Sistema de Alertas e Notificações

Por quê: Notificações impulsionam ações. Sem elas, tarefas caem no limbo.

O que implementar:

  • Modelo Notification: userId, type, message, isRead, relatedEntityId, createdAt
  • Sistema de eventos: Usar @nestjs/event-emitter para disparar notificações em eventos (OS criada, pagamento recebido, garantia expirando)
  • UI: Bell icon no header, dropdown com notificações recentes, mark as read
  • Canais: In-app (banco de dados) + Email + Push (Service Worker)
  • Templates: Diferentes mensagens por tipo de evento

Esforço: 18-24 horas | Banco: 1-2 novos modelos

ALTO Semana 5-6

7. Assinatura Digital e Aceite no Local

Por quê: Prova de entrega reduz disputas. Assinatura digital do cliente no tablet/mobile é proof-of-work.

O que implementar:

  • Modelo SignatureRequest: serviceOrderId, signatureData (SVG), signerName, signedAt, ipAddress
  • Widget: Canvas para desenhar assinatura no mobile/tablet
  • Workflow: Ao clicar "Finalizar Trabalho", exibir tela de assinatura
  • PDF: Gerar recibo com assinatura embedded (já tem PdfModule)
  • Versão zero-cost: Usar canvas nativo + PDF gerado com pdfkit (npm package)

Esforço: 16-20 horas | Banco: 1 novo modelo

ALTO Semana 6-7

8. Integração PIX / Conciliação Bancária

Por quê: PIX é padrão no Brasil. Confirmar pagamentos automáticamente reduz trabalho manual.

O que implementar:

  • Gerar QR Code PIX: Para cada pagamento, gerar QR dinâmico (brcode) via Asaas ou Stripe
  • Webhook de confirmação: Provider envia POST quando PIX é pago → atualizar status Payment para PAID
  • Relatório: Extrato diário de PIX recebidos vs OS completadas
  • Providers zero-cost: Asaas (free até 500 transações) ou Stripe (2.9% + 0.30)
  • UI: Mostrar QR code na tela de pagamento, histórico de transações

Esforço: 25-35 horas | Banco: 0 novos modelos (adicionar campos em Payment)

MÉDIO Semana 7-8

9. Avaliação e Satisfação de Cliente

Por quê: Feedback contínuo melhora qualidade. NPS ajuda a identificar problemas sistemáticos.

O que implementar:

  • Modelos: Review (serviceOrderId, rating 1-5, comment, createdAt), NPS (surveyId, clientId, score)
  • Trigger: Enviar link de avaliação por email 1 dia após OS completada
  • Dashboard: Rating médio por técnico, comentários recentes, NPS trend
  • Alertas: Se rating < 3, notificar gerente para follow-up
  • Integração: Relacionar reviews com técnico para identificar problemas

Esforço: 14-18 horas | Banco: 2 novos modelos

MÉDIO Semana 8-9

10. Relatórios Exportáveis (PDF, Excel, CSV)

Por quê: Exportação simplifica envio de dados para contador, gestor, ou análise externa.

O que implementar:

  • Relatórios: Fluxo de caixa, receita por técnico, serviços mais vendidos, histórico de garantias
  • Formatos: PDF (pdfkit), Excel (xlsx), CSV
  • Filtros: Por período, técnico, serviço, cliente
  • Agendamento: Enviar relatório automático por email (segunda-feira 8h, por exemplo)
  • UI: Botão "Exportar" em cada relatório, preview antes de download

Esforço: 16-22 horas | Banco: 0 novos modelos (usar dados existentes)

MÉDIO Semana 9-10

11. Templates de OS e Documentos Customizáveis

Por quê: Documentos profissionais reforçam marca. Economia de tempo na repetição.

O que implementar:

  • Modelos: Template (name, content, type: OS/QUOTE/INVOICE), TemplateVariable (templateId, variable, label)
  • Editor: Interface WYSIWYG para criar templates, preview com variáveis substituídas
  • Uso: Ao criar OS, selecionar template → preencher variáveis → gerar documento
  • Defaults: Incluir 3-4 templates padrão no setup
  • Exportação: Gerar PDF/Word de OS com template aplicado

Esforço: 18-25 horas | Banco: 2 novos modelos

MÉDIO Semana 10-11

12. Gestão de Fornecedores e Compras

Por quê: Centralizar contatos de fornecedores. Rastrear pedidos. Comparar preços.

O que implementar:

  • Modelos: Supplier (name, phone, email, address, rating), PurchaseOrder (supplierId, items, totalValue, status, deliveryDate)
  • CRUD: Gerenciar fornecedores, criar pedidos, rastrear entrega
  • Integração: Vincular PurchaseOrder a MaterialUsage (quando material é consumido em OS)
  • Relatório: Custo de materiais por fornecedor, comparativo de preços
  • Alertas: Notificar quando estoque baixo (disparar compra automática)

Esforço: 20-28 horas | Banco: 2-3 novos modelos

BAIXO Semana 11-12

13. App Mobile Nativo (React Native / Expo)

Por quê: Técnico precisa acessar agenda e registrar trabalho no local. PWA não substitui app.

O que implementar:

  • Stack: React Native + Expo (multiplataforma Android/iOS, zero custo de build)
  • Funcionalidades: Agenda do dia, iniciar/finalizar OS, capturar fotos, assinar, offline-first
  • Offline: Usar WatermelonDB ou Realm para sincronizar quando online
  • Geolocalização: Mapear técnicos em tempo real (opcional: integrar Google Maps)
  • Biometria: Login por face/fingerprint

Esforço: 40-60 horas | Banco: 0 novos modelos (reutilizar APIs)

BAIXO Semana 12+

14. IA para Estimativa de Preço e Categorização

Por quç: Automação de reparos similares. Sugestão de preço baseado em histórico.

O que implementar:

  • Modelo de ML: Treinar com histórico de OS (descrição → categoria → preço médio)
  • API: Usar Google Gemini (free tier) ou OpenAI (GPT-4) para análise de descrição
  • Workflow: Usuário escreve descrição → IA sugere categoria, serviços, preço → confirmar e criar OS
  • Histórico: Rastrear acurácia da sugestão, reajustar modelo
  • Chatbot: Assistente para responder perguntas sobre disponibilidade, preço, prazo

Esforço: 30-50 horas | Banco: 1 novo modelo (AIInteraction log)

  • Integração WhatsApp / SMS para Notificações

  • Controle de Inventário (Materiais)

🟢 FASE 4: Qualidade e Escala (Semanas 7-9) - Crescimento
  • Avaliação e Satisfação de Cliente
  • Relatórios Exportáveis (PDF, Excel, CSV)

Impacto: Qualidade melhora. Documentos profissionais. Dados prontos para análise externa (contador, gestor).

  • Gestão de Fornecedores e Compras

⚪ FASE 6: Canais Digitais (Semanas 12+) - Ambição
  • App Mobile Nativo (React Native / Expo)
  • IA para Estimativa de Preço e Categorização

Impacto: App standalone. Automação inteligente. Escalabilidade para múltiplos técnicos/regiões.

⏱️ Timeline Total: 12 semanas (~3 meses) para implementar tudo. Sequência é flexível: se faltam fontes no primeiro mês, pule app mobile para frente. Se precisa de receita imediata, foque em PIX + Dashboard primeiro.
